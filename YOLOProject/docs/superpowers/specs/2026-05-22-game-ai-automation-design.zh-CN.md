# 游戏 AI 自动化系统设计

## 背景

当前自动化平台已经支持项目、环境、用例集选择，支持选择执行设备，支持把任务分发到多台机器，并能长时间运行稳定性任务。现有系统已经可以检测游戏是否卡死、宕机或异常退出。

当前主要痛点是：

- 现有用例是固定的结构化步骤序列。节点、UI、流程或游戏状态一变化，用例就容易跑不起来。
- 系统能判断游戏是否还活着，但不能可靠判断游戏功能是否正确。
- 用例编写成本高。每个用例都需要人工组合游戏节点。
- 希望系统通过长期执行和学习，逐步总结游戏模块、玩法和任务，并自动生成更准确的功能测试用例。

本设计的方向是增强现有平台，而不是重做调度系统。

后续工程落地仓库：

- https://github.com/caiqifeng/MyTestTool

## 目标

1. 使用 YOLO26、OCR、截图和时序视觉证据，为现有用例增加黑盒功能验证能力。
2. 从历史结构化用例和游戏节点/API 能力库中生成测试用例草稿。
3. 自动生成用例时产出三份结果：现有平台结构化步骤、行为树草稿、断言配置。
4. 从大规模机器长期执行中持续学习，提高测试覆盖率和用例生成质量。
5. 复用现有调度、设备分发和长时间执行能力。

## 非目标

- 不重做当前任务调度平台。
- 不替换当前执行设备管理系统。
- MVP 阶段不做完全无人审核的自动用例生成。
- 第一阶段不实现完整动态执行。
- 不依赖单帧截图直接判断功能正确。
- MVP 阶段不做完整行为树可视化编辑器。

## 总体方向

系统应从“固定步骤执行”演进为“测试意图 + 游戏状态 + 功能断言”驱动的自动化层。

当前用例模型：

- 第一步、第二步、第三步、第四步是固定的。
- 只有当预期节点顺序仍然匹配当前游戏时，用例才能稳定执行。

目标用例模型：

- 用例有明确测试意图，例如验证任务完成、奖励领取。
- YOLO26/OCR 识别当前游戏状态。
- 断言系统验证功能结果是否真正发生。
- 历史步骤和节点元数据用于生成新用例草稿。
- 行为树草稿保存状态转移和恢复逻辑，为后续动态执行做准备。

## 优先级

增强路线按以下顺序推进：

1. 功能验证。
2. 自动生成用例。
3. 动态执行。

功能验证必须优先，因为如果系统不能判断功能结果是否正确，生成再多用例也没有测试价值。

## 现有平台接入

现有平台继续负责：

- 项目选择。
- 环境选择。
- 用例集选择。
- 执行设备选择。
- 多机器任务分发。
- 长时间执行。
- 稳定性检测，包括卡死、崩溃和异常退出。

新增系统负责增强：

- 功能验证插件。
- YOLO26/OCR 检测层。
- 断言引擎。
- 证据报告扩展。
- 节点能力库。
- 用例生成助手。
- 行为树草稿输出。
- 基于执行轨迹的持续学习流水线。

## 功能验证

### 验证模式

由于当前暂时拿不到任务状态和奖励到账的日志/API，MVP 默认按黑盒验证设计。

系统需要支持：

- YOLO26 视觉目标检测。
- OCR 文本识别。
- 截图证据。
- 多帧时序证据。
- Pass / Fail / Unknown 三态结果。

必须支持 Unknown。视觉证据不足时，系统不能误判为通过。

### 第一批检测范围

第一优先级标签：

- `quest_tracker`
- `quest_complete_hint`
- `reward_popup`
- `claim_reward_button`
- `reward_text_area`
- `error_popup`
- `network_error`
- `blocked_flow_hint`

这些标签优先覆盖最关键的问题：任务、奖励和异常是否能被检测到。

第二优先级标签：

- 登录/主界面。
- NPC 对话/确认按钮。
- 战斗/结算状态。

### 断言模型

断言应基于状态变化，而不是单帧截图。

任务完成断言示例：

- 执行前检测任务追踪栏。
- 执行后检测任务进度变化或任务完成提示。
- 可选 OCR 识别任务文本或完成提示文本。
- 检测奖励弹窗或领取结果。
- 在多帧画面中确认预期状态持续出现。
- 保存截图、检测框、OCR 文本、时间戳和置信度。

### 模型选择

MVP 使用 YOLO26 作为默认视觉检测基线。

工程接口仍然必须使用 Detector 抽象：

- `YOLO26Detector`
- `OCRDetector`
- `TemplateDetector`
- `DetectorAdapter`

统一输出结构：

- `label`
- `bbox`
- `confidence`
- `text`
- `timestamp`
- `source`

模型版本、数据集版本、阈值和评估结果必须绑定到项目/游戏版本，并写入执行报告。

## 数据资产

团队已经有大量截图和录像。

MVP 数据处理流水线：

1. 收集历史截图、录像和失败画面。
2. 从录像中抽帧。
3. 清洗和去重图片。
4. 归一化分辨率，并标记游戏版本和环境。
5. 定义任务、奖励、异常相关标签。
6. 人工标注第一批数据集。
7. 划分训练集、验证集和测试集。
8. 使用历史录像回放评估 YOLO26 效果。

## 自动生成用例

### 来源优先级

采用混合来源，但 MVP 优先使用：

1. 历史结构化用例。
2. 游戏节点/API 能力库。

后续阶段加入：

3. 长期执行轨迹。
4. 需求文档、策划文档、玩法说明和版本变更文档。

### 节点能力库

当前系统可以提供：

- 节点名。
- 所属模块/玩法。
- 输入参数。
- 前置条件。
- 执行动作。
- 成功后状态。
- 失败后状态。

缺少字段：

- 可用环境。
- 风险等级。
- 可用于哪些断言。

这些缺失字段先由系统推断，再在人工审核中修正。

推断示例：

- 可用环境：根据项目、版本、环境标签和历史执行成功率推断。
- 风险等级：根据是否消耗资源、是否改变账号状态、是否影响经济系统、是否允许在线上环境执行来推断。
- 可用于哪些断言：根据成功后状态、YOLO/OCR 观察和可用证据推断。

### 生成用例输出

每个自动生成的用例应产出三份结果：

1. 现有平台结构化步骤。
2. 行为树草稿。
3. 断言配置。

结构化步骤用于兼容现有平台。行为树草稿保存测试意图、状态检查、节点选择和恢复策略，为后续动态执行做准备。断言配置定义 YOLO26/OCR/截图时序检查规则。

生成用例必须包含：

- 用例目标。
- 适用项目、版本和环境。
- 账号需求。
- 步骤列表和参数。
- 预期状态。
- 功能断言。
- 恢复策略。
- 风险等级。
- 生成依据。

### 审核流程

生成用例先进入草稿状态。

流程：

1. 生成草稿。
2. 执行风险检查。
3. 人工审核。
4. 试运行。
5. 提升为正式用例集。

MVP 阶段，自动生成用例不能未经审核直接进入正式回归。至少需要人工确认，并完成一次成功试运行。

## 持续学习

系统应利用大量机器长期执行，不断提高测试准确性。

持续学习的主要目标不是“玩得更强”，而是“测得更准”。

执行机器池分为：

- 探索池：开发服/测试服，允许执行更宽松的学习任务。
- 回归池：执行已审核用例。
- 验收池：只执行受控用例。

学习产出：

- 更准确的功能断言。
- 更稳定的 UI/状态识别。
- 新用例建议。
- 缺失覆盖发现。
- 失败模式总结。
- 节点和断言置信度更新。

系统必须区分：

- 观察到的执行轨迹。
- 推断出的玩法/模块模型。
- 已审核的正式测试用例。

## 行为树的角色

行为树不是当前平台第一阶段必须直接执行的格式，但它是重要的中间表达。

MVP 应生成行为树草稿，用来表达：

- 测试意图。
- 状态检查。
- 节点替代路径。
- 失败恢复。
- 功能断言。

后续动态执行可以基于这些行为树草稿，逐步摆脱固定步骤顺序。

## 报告

报告需要扩展当前结果模型。

每个功能验证结果应包含：

- Pass / Fail / Unknown。
- 断言发生在哪个步骤。
- 截图。
- YOLO 检测框。
- OCR 结果。
- 置信度。
- 时序证据帧。
- 失败或 Unknown 原因。
- 模型版本。
- 数据集版本。
- 如果是生成用例，记录用例生成来源。

报告应能说明失败原因属于：

- 功能结果没有出现。
- 视觉证据不足。
- OCR 不确定。
- 出现异常 UI。
- 现有执行流程在到达验证点前失败。

## MVP 里程碑

### M0：功能验证原型

- 建立 Detector 抽象。
- 接入 YOLO26 截图推理。
- 接入 OCR。
- 定义第一批标签。
- 在历史截图/录像上运行检测。
- 对任务、奖励、异常样本输出 Pass / Fail / Unknown 断言结果。

### M1：接入现有用例验证

- 给现有结构化用例附加断言配置。
- 通过现有平台执行当前用例。
- 在报告中增加功能证据。
- 支持任务、奖励、异常检测。

### M2：用例生成草稿

- 导入历史结构化用例。
- 导入节点能力库。
- 生成新的结构化用例草稿。
- 生成行为树草稿。
- 生成断言配置。
- 增加人工审核和试运行流程。

### M3：持续学习闭环

- 收集执行轨迹。
- 总结成功/失败模式。
- 推断缺失节点字段。
- 建议新用例和缺失断言。
- 将审核结果回流到节点能力库。
